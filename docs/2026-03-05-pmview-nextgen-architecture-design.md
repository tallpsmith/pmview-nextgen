# pmview-nextgen Architecture Design

**Date:** 2026-03-05
**Status:** Approved
**Approach:** B — End-to-End Spike First, then TDD

## Vision

Bridge PCP performance metrics into living, breathing 3D Godot scenes. An SRE
on their Mac or Linux desktop points pmview-nextgen at a remote pmproxy endpoint
and watches their server farm come alive.

## Architecture Overview

Three layers, each independently testable and deployable:

```
+-----------------------------------------------------+
|                   Godot Engine                        |
|                                                       |
|  +---------------+    +---------------------------+  |
|  |  GDScript     |    |   C# Bridge Layer         |  |
|  |  Scene API    |<---|   (GodotPcpBridge.cs)      |  |
|  |  (scenes)     |    |   - metric polling loop    |  |
|  +---------------+    |   - value normalization    |  |
|                       |   - scene binding config   |  |
|                       +-------------+-------------+  |
+------------------------------------|------------------+
                                     | uses
                       +-------------v--------------+
                       |  PcpClient (.NET library)   |
                       |  - HTTP client for pmproxy  |
                       |  - metric discovery         |
                       |  - fetch / subscribe        |
                       |  - fully TDD, no Godot dep  |
                       +-------------+--------------+
                                     | REST/HTTP
                       +-------------v--------------+
                       |  pmproxy (podman container) |
                       |  + valkey + PCP pmlogger    |
                       |  + pmlogsynth test data     |
                       +-----------------------------+
```

## Key Decisions

### C# for the PCP Client Library

The PcpClient library is a standalone .NET class library (net8.0+) with no Godot
dependency. This gives us:

- xUnit/NUnit testing without the Godot runtime
- Clean separation: the library knows about PCP, not about Godot or scenes
- Future independence: when it earns its own repo, it becomes a NuGet package
  any .NET project can consume

The library starts inside this repo (`src/pcp-client-dotnet/`) and will be split
into its own repository when it matures beyond pmview-nextgen's immediate needs.

### Cross-Platform from Day One

- Primary audience: Linux (where PCP is native)
- Development environment: macOS
- .NET targets a recent LTS that runs on both
- Godot 4.x is well-supported on both platforms
- Validate Linux packaging early, not as an afterthought

### Network Connectivity Model

pmproxy is always a network endpoint. The PcpClient accepts a base URL at
construction time (`http://host:44322`). "Local" is just `localhost` — there is
no special local mode. Primary real-world use: SRE's desktop connecting to a
remote pmproxy on a collector host.

### AI World Crafting (godot-mcp)

The godot-mcp server (https://github.com/Coding-Solo/godot-mcp) serves dual
purpose:

1. **Near-term:** Development accelerator for Claude Code to rapidly craft and
   iterate on Godot scenes without manual editor work.
2. **Long-term:** Proof of concept for a product feature — "AI World Creator"
   where users describe visualisation worlds in natural language and the system
   generates Godot scenes with metric bindings.

Spike 3 validates whether this is viable.

## Dev Environment

Podman compose stack adapted from the pmmcp project pattern
(https://github.com/tallpsmith/pmmcp):

```
dev-environment/
  compose.yml
  pmlogsynth/
    configs/          # Synthetic data generation configs
  README.md
```

| Service  | Purpose                                       |
|----------|-----------------------------------------------|
| pcp      | PCP collector (pmcd), ingests synthetic data   |
| pmproxy  | REST API gateway, serves metrics over HTTP     |
| valkey   | Required by pmproxy for time-series queries    |

Synthetic data generated by pmlogsynth
(https://github.com/tallpsmith/pmlogsynth), ingested on container startup.
Deterministic and reproducible — replay any workload profile on demand.

This stack is shared across all spikes and persists into production development.

## Prototype Sequence

Constitution mandates prototype-first validation. Three spikes, each with a
clear completion gate. Prototype code does not graduate to production.

### Spike 1: Connectivity Proof

**Goal:** Prove C# inside Godot can fetch a real metric from pmproxy.

- Stand up the podman dev stack
- Ingest synthetic data via pmlogsynth
- Write minimal C# in Godot: HttpClient -> GET pmproxy -> parse JSON -> print
- **Done when:** A real metric value prints to Godot's console.

### Spike 2: Scene Binding Proof

**Goal:** Prove real PCP data can drive a 3D scene.

- Extend the crude C# script to poll pmproxy on a timer
- Wire one metric (kernel.all.load) into the existing pmview_classic scene's
  set_load_average() API
- Replace one sine wave with reality
- **Done when:** A 3D object moves in response to real (synthetic) PCP data.

### Spike 3: AI World Crafting Proof (can overlap with Spike 2)

**Goal:** Evaluate godot-mcp for scene generation via natural language.

- Install and configure godot-mcp
- Attempt to describe and generate a simple scene
- Evaluate output quality, cleanup effort, and viability as a product feature
- **Done when:** Honest assessment of godot-mcp capability and go/no-go decision.

### After the Spikes

With all three validated, we know the full pipe works. Then:

1. TDD the proper PcpClient library, informed by real pmproxy API behaviour
2. Build the C# Bridge Layer in Godot
3. Design the metric binding configuration system
4. Craft production-quality scenes

## C# Library Scope (Initial)

Start with what the spikes need. Grow from there.

- **Context** — create/destroy metric collection contexts
- **Fetch** — get current values for named metrics
- **Describe** — discover available metrics, types, semantics
- **Indom** — instance domain enumeration (per-CPU, per-disk, etc.)

The library is true to PCP's domain model. It knows about PCP concepts — not
about Godot, scenes, or pmview-nextgen. This boundary is deliberate: when the
library earns its own repo, it should be useful to any .NET project working
with PCP.

## Open Questions (to be resolved by spikes)

- What pmproxy API quirks will we discover? (Spike 1)
- What polling frequency is practical without jitter? (Spike 2)
- Is godot-mcp mature enough to invest in? (Spike 3)
- What .NET version gives us the best cross-platform story?
- What does the metric binding config format look like? (post-spike design)
